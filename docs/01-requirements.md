# 01. 需求 (Requirements)

本文件記錄體能訓練週期課表管理系統在「需求」階段的工作過程：如何接到需求、辨識誰是利害關係人、用哪些手法把需求引出來、如何把口語訴求釐清成可動工的內容，最後彙整為設計階段使用的情境。

---

## 1.1 需求受理 Intake

### 需求來源

需求由一位**體能教練（同時為工作室經營者）** 主動提出，背景是他過去長期以 Google Sheets 管理週期課表，並另外登入 VALD Performance 後台讀取檢測數據。隨著合作的學員規模成長至 50 人以上（涵蓋職業選手、校隊與業餘三類），既有工具的維護成本不再可控，因此提出一套整合性系統的開發需求。

### 受理渠道

需求並非以單次正式書面 RFP 形式提出，而是以下列三種渠道交織進行：

| 渠道                | 用途                                                                   |
| ------------------- | ---------------------------------------------------------------------- |
| 口頭 / 面對面討論   | 首次接洽、整體目標對齊、關鍵決策（如資料邊界、模板與排程拆分）         |
| 即時通訊(Instagram) | 日常細節澄清、補充情境、變更需求、回報問題                             |
| 現場操作觀察        | 請教練在實際工作環境中走一遍排課流程與 VALD 數據處理，由 SA 旁觀並紀錄 |

### 初版需求摘要

口語版需求經整理後可歸納為一句話：

> 「讓我可以在同一個地方建立課表、安排每位學員的訓練日期，並把 VALD 的檢測資料一起看，不要每次都在不同 Sheet 與 VALD 後台之間切來切去。」

### 優先級評估

由於需求方為單一決策者（工作室老闆）。實務上依下列兩個維度排序：

| 維度       | 說明                                                                                           |
| ---------- | ---------------------------------------------------------------------------------------------- |
| 操作頻率   | 教練每日 / 每週都要做的動作（排課、查 VALD），優先處理                                         |
| 痛點影響面 | 影響到學員體驗的（手機看 Sheet 不友善），與影響到歷史資料正確性的（VALD 資料手抄），列為第一波 |

最終第一波範圍鎖定在三大情境：**課表管理、VALD 整合、學員端今日訓練**。其餘（如跑步 / 體能手動資料、衍生指標延伸）列入後續迭代。

## 1.2 利害關係人分析

### 角色盤點

| 利害關係人                 | 角色定位                   | 與系統互動方式                                                     |
| -------------------------- | -------------------------- | ------------------------------------------------------------------ |
| 工作室老闆 / 主教練        | 需求方、決策者、主要使用者 | 建立課表模板、為學員排課、同步 VALD 資料、查看檢測趨勢             |
| 學員（職業 / 校隊 / 業餘） | 終端使用者                 | 登入後查看自己的今日訓練與檢測歷史                                 |
| 工作室其他教練             | 潛在後續使用者，本期不納入 | 不在第一版範圍內；架構保留多教練擴充可能                           |
| VALD Performance           | 第三方資料來源             | 透過 OAuth2 Client Credentials 提供 Profile 與 ForceDecks 檢測資料 |
| SA / 開發團隊              | 交付方                     | 需求釐清、建模、規格、實作與部署                                   |

### RACI

針對主要 SA 階段交付物：

| 交付物              | 工作室老闆 | SA / 開發團隊 | VALD（外部）  |
| ------------------- | ---------- | ------------- | ------------- |
| 需求 / 情境定義     | A          | R             | C             |
| Use Case / 流程設計 | C          | R / A         | I             |
| 資料模型            | I          | R / A         | —             |
| VALD 整合策略       | C          | R / A         | C（API 規格） |
| 驗收 / UAT          | A          | R             | —             |

> 註：R=Responsible, A=Accountable, C=Consulted, I=Informed。學員未直接參與決策；其需求由教練代述，並於上線後以實際使用回饋驗證。

## 1.3 需求引導 Elicitation

考量需求方僅有教練一人、需求又分散在日常聊天中，採用以下三種互補手法：

### 手法 A — 面對面訪談

於工作室現場進行多次訪談，主軸為理解：

| 主題         | 主要詢問                                                  |
| ------------ | --------------------------------------------------------- |
| 既有工作流   | 一份新課表從零到上線給學員看，要經過哪些步驟？            |
| 既有工具痛點 | 哪些動作每週重複做？哪些步驟最容易出錯？                  |
| 課表分類     | 課表有哪些型態？有沒有固定不變的 vs 週期變化的？          |
| 學員端使用   | 學員在什麼情境下看課表？通常用什麼裝置？                  |
| 資料邊界     | VALD 資料與本地資料的關係要走哪個方向？是否需要雙向同步？ |

### 手法 B — 現場操作觀察

口頭描述容易遺漏實際操作中的隱性步驟，因此請教練在 SA 在場時實際操作一遍：

1. 從 Google Sheets 範本複製一份新的學員課表
2. 依照學員當前狀況調整動作、組數、強度
3. 登入 VALD 後台，逐一查詢該學員的 SJ / CMJ / IMTP / HJ 檢測資料
4. 將常用指標手抄回 Sheet，並手動計算 DSI / EUR
5. 將安排好的課表透過手機展示給學員看

此過程揭露了多項口頭訪談中未提到的痛點，包括：手機橫向拖移 Sheet、同一天多種訓練類型分散在不同分頁、手抄資料容易輸錯等。

### 手法 C — 即時通訊持續澄清

主要訪談結束後，需求並未一次定案。後續開發過程中透過 Instagram 持續溝通，常見訊息類型：

- 補充上次訪談沒講到的細節
- 提出新的使用情境（例如固定暖身課表）
- 對既有功能的調整建議（例如固定課表日期選擇麻煩）
- 回報實際使用中發現的問題

這部分訊息經整理後成為**變更管理**（見 [05-governance.md](05-governance.md) 5.3 節）的主要輸入。

## 1.4 需求釐清 Clarification

教練的需求多以「現場語言」表達，並常夾帶他自己的解法。SA 在此階段的工作不是直接接受，而是反覆釐清「想要的」與「真正需要的」是否一致。常用的釐清手法：

### 手法：將表面解法回推到根本痛點

任何「我想要 X 功能」的訴求，先回問「為什麼需要 X？X 解決了什麼問題？」往往會發現真正的需求和最初表述不同。

### 釐清紀錄（節選）

| 教練口語訴求                 | 5 Why 後的真實需求                                       | 採用做法                                                                      |
| ---------------------------- | -------------------------------------------------------- | ----------------------------------------------------------------------------- |
| 「我要一個固定課表的新功能」 | 並非結構不同，而是「不需要週次區分的同質訓練內容」       | 沿用既有 TrainingCycle 結構，以 `WeekNumber = 0` 表示固定課表，避免新增資料表 |
| 「固定課表不用選日期」       | 並非真的不要日期，而是「日期區間通常很長、手動選太麻煩」 | 保留 `StartDate` / `EndDate`，前端加入 3 / 6 / 12 個月快選                    |

> 兩個完整的釐清案例（固定課表資料結構、固定課表日期選擇）以變更管理形式記錄於 [05-governance.md](05-governance.md) 5.3 節。

### 釐清的價值

從上表可看出：**若直接照字面實作教練的訴求，至少會多出 2 張新資料表、3 套並行邏輯、以及一組無日期邊界的特殊規則**。釐清的目的不是質疑教練，而是把口語需求對齊到既有資料模型可承接的位置，降低長期維護成本。

## 1.5 需求情境彙整

將前述受理、訪談、觀察與釐清結果，彙整為三組情境，作為下一階段（02-analysis、03-design）的基礎。

### 情境 1 — 排課資料分散

| 項目     | 內容                                                                                                         |
| -------- | ------------------------------------------------------------------------------------------------------------ |
| 觀察     | 教練在 Google Sheets 維護多份範本，排課時複製貼上、調整日期與動作                                            |
| 痛點     | 範本散落、版本不一致；單人排課約需 45 分鐘                                                                   |
| 需求轉譯 | 系統需提供可重複使用的**課表模板**；模板套用後自動建立**獨立學員副本**；課表需支援「固定」與「週期」兩種模式 |
| 設計鉤點 | `TrainingCycle.UserId = null` 表示模板；有值表示學員專屬副本；`WeekNumber = 0` 表示固定課表                  |

### 情境 2 — VALD 數據查詢與手抄耗時

| 項目     | 內容                                                                                                                                 |
| -------- | ------------------------------------------------------------------------------------------------------------------------------------ |
| 觀察     | 教練登入 VALD 後台逐一查詢 SJ / CMJ / IMTP / HJ，手抄常用指標到 Sheet，並以公式計算 DSI / EUR                                        |
| 痛點     | 手抄易錯；VALD 後台查詢慢；歷史資料無法集中追蹤                                                                                      |
| 需求轉譯 | 系統需以**單向匯入**方式同步 VALD Profile 與 Tests；本地保存**檢測快照**；DSI / EUR 由系統**自動計算**；歷史查詢不依賴第三方即時 API |
| 設計鉤點 | `ValdProfileId` 綁定本地使用者；`ModifiedFromUtc` 增量同步；本地 `vald_test_metric_records` 快照                                     |

### 情境 3 — 學員手機查看課表不友善

| 項目     | 內容                                                                                                                               |
| -------- | ---------------------------------------------------------------------------------------------------------------------------------- |
| 觀察     | 學員多在訓練現場用手機開啟 Sheet；需橫向拖移才能看完整欄位；同一天的核心 / 活動度 / 肌力訓練分散在不同分頁                         |
| 痛點     | 訓練中斷感明顯、查資料效率低、教練還需個別講解順序                                                                                 |
| 需求轉譯 | 系統需提供**手機優先的學員端**；首頁直接呈現**今日訓練**；同一天多種訓練類型**依排程順序整合於同一頁**；動作資訊以**直向卡片**呈現 |
| 設計鉤點 | 學員端以 `UserId` 查詢 `AthleteSchedule`，依 `Order` 整合展開 `TrainingCycleWeek` → `Group` → `Item`                               |

### 情境間的依賴關係

```mermaid
flowchart LR
    S1["情境 1<br/>課表模板與排程"]
    S2["情境 2<br/>VALD 整合"]
    S3["情境 3<br/>學員端今日訓練"]

    S1 --> S3
    S2 -. 提供決策依據 .-> S1

    classDef base fill:#F4F6FA,stroke:#5B8DEF,color:#1F3A68;
    class S1,S2,S3 base;
```

情境 1 是核心：沒有課表與排程，情境 3 沒有資料來源。情境 2 雖獨立運作，但實務上 VALD 檢測結果會回饋到教練如何排下一個週期，因此屬於支援性需求。
