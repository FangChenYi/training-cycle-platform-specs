# 05. 治理 (Governance)

本文件記錄體能訓練週期課表管理系統在治理層面的工作方式：評審 / 簽核如何進行、用什麼作為 Baseline、需求變更怎麼管、UAT 與結案如何收尾。

本案的決策結構單純（教練 = 工作室老闆 = 唯一需求方），不適合套用大型專案的正式治理儀式；本文件採「以實質儀式取代正式儀式」的原則記錄。

---

## 5.1 評審簽核

### 決策結構

依 [01-requirements.md § 1.2 RACI](01-requirements.md) 的劃分，本案只有兩個治理角色：

| 角色                | 決策權                                       |
| ------------------- | -------------------------------------------- |
| 工作室老闆 / 主教練 | 需求方、最終 Accountable；範圍與優先級拍板者 |
| SA / 開發團隊       | Responsible；提案、設計、實作、驗證          |

### 評審方式

不採正式評審會議與書面簽核；以下列實質審議取代：

| 階段        | 實質審議方式                                                          | 對應的「簽核」                                            |
| ----------- | --------------------------------------------------------------------- | --------------------------------------------------------- |
| 需求對齊    | 面對面訪談 + 現場操作觀察                                             | 教練口頭確認；SA 將結論寫回 [01](01-requirements.md)      |
| 範圍 / 設計 | 重大決策（如資料邊界、模板與固定課表的承接方式）由 SA 提案 → 教練拍板 | 決策落地為 [02](02-analysis.md) / [03](03-design.md) 章節 |
| 規格細節    | 透過實際操作驗證；不正確 / 不順手即列為變更輸入                       | 教練實際使用 = 隱性簽核                                   |

> 此模式可行的前提是「決策者就是日常使用者」，能即時用使用感受驗證設計是否落地。若未來擴張為多教練 / 多工作室，需重新評估是否導入正式評審。

## 5.2 Baseline

本案未做正式版本凍結。實務上以 git 上的最近一次 commit 作為實質基準，5.3 變更管理的「從哪裡算起」即以此為錨。若未來導入正式評審或多人決策，再考慮引入版本凍結式 baseline。

## 5.3 變更管理

### 規則

| 規則               | 採用做法                                                 |
| ------------------ | -------------------------------------------------------- |
| 哪些動作算「變更」 | 既有 baseline 後，對需求 / 範圍 / 設計 / 規格的修改      |
| 變更觸發來源       | 教練回饋、開發過程中發現的不合理                         |
| 變更紀錄格式       | 每筆一個 CR-編號 + 觸發 / 評估方案 / 採用做法 / 影響範圍 |

### 變更清單

| 編號            | 標題                            | 觸發來源                      | 採用做法摘要                                                                                                  |
| --------------- | ------------------------------- | ----------------------------- | ------------------------------------------------------------------------------------------------------------- |
| [CR-01](#cr-01) | 新增「固定課表」類型            | SA 觀察既有 Sheet 時主動發現  | 不另建表，沿用週期課表結構承接，以「週次 = 0」表示固定                                                        |
| [CR-02](#cr-02) | 固定課表日期需求調整            | 教練回饋「每次選日期很繁瑣」  | 保留日期欄位，前端加入 3 / 6 / 12 個月快選按鈕                                                                |
| [CR-03](#cr-03) | VALD Profile 同步：從雙向到單向 | VALD 端限制讓雙向同步無法落地 | 改採單向匯入；詳見 [02-analysis.md § VALD Profile 同步策略](02-analysis.md#vald-profile-同步策略從雙向到單向) |

### <a id="cr-01"></a>CR-01 新增「固定課表」類型

| 項目         | 內容                                                                                                            |
| ------------ | --------------------------------------------------------------------------------------------------------------- |
| 觸發         | 觀察教練既有 Sheet 時，發現課表並非單一型態：除了需週次切換的週期課表外，還有同一份內容反覆使用、不需週次的型態 |
| 釐清過程     | 向教練確認後，確立「固定課表」（每次來都練一樣的內容，適合暖身、活動度）為與週期課表並列的新型別                |
| 原始狀態     | 系統最初只規劃週期課表                                                                                          |
| 評估方案     | (A) 為固定課表新建 table 與獨立邏輯；(B) 沿用週期課表結構，以特殊值表示「固定」                                 |
| 採用做法     | (B)：固定課表 = 一份只有一週的週期課表，該週「週次」以 `0` 表示                                                 |
| 為何不選 (A) | 新建 table 會帶來並行的查詢 / 排程 / UI 邏輯；對既有結構與前端衝擊都大                                          |
| 影響範圍     | 資料層：無新表、無 schema 變更；後端：放寬「同一份課表內可有一週 0 號週」；前端：對「週次 = 0」做呈現分支       |
| 後續對應     | [03-design.md § ER 關鍵設計取捨](03-design.md#關鍵設計取捨)：「以週次 = 0 表示固定課表」                        |

### <a id="cr-02"></a>CR-02 固定課表日期需求調整

| 項目         | 內容                                                                                        |
| ------------ | ------------------------------------------------------------------------------------------- |
| 觸發         | 教練回饋：「固定課表每次都要選日期很繁瑣，希望乾脆不用選」                                  |
| 釐清過程     | 觀察既有資料 → 固定課表仍有 1~2 個月的日期區間；追問後教練表示「會依選手賽季調整時間」      |
| 真實需求     | 不是不要日期，是「日期區間長、手動選太麻煩」                                                |
| 評估方案     | (A) 取消固定課表的日期欄位；(B) 保留日期，UI 加入快選機制                                   |
| 採用做法     | (B)：保留起訖日期，前端加入「3 / 6 / 12 個月」快選按鈕                                      |
| 為何不選 (A) | 取消日期會破壞既有排程的區間校驗；未來修改課表內容會回頭影響過去的排程，造成資料汙染        |
| 影響範圍     | UI：新增快選元件；後端與資料層：不動                                                        |
| 後續對應     | [01-requirements.md § 1.4 釐清紀錄](01-requirements.md)：「固定課表不用選日期」對應該筆釐清 |

## 5.4 UAT

### UAT 方式

不安排獨立的 UAT 場次；以教練實際使用作為驗收：

| 項目     | 採用做法                                                                             |
| -------- | ------------------------------------------------------------------------------------ |
| 驗收對象 | 第一波範圍三大情境（課表管理、VALD 整合、學員端今日訓練）                            |
| 驗收方式 | 教練在工作室實際排課、實際同步 VALD、實際讓學員用手機開課表                          |
| 驗收標準 | 對照 [04 規格 § 驗收準則](04-specification.md#41-功能規格-fsd)；不通過即記為變更輸入 |
| 回饋管道 | 面對面（工作室碰面）、IG 即時通訊                                                    |
| 處理節奏 | 回饋累積到一定數量即消化一輪，依重要性排入下一波開發                                 |

### 學員端驗收

學員未直接參與決策，採「教練代述 + 上線後實際使用回饋」的方式驗證：

- 教練在現場觀察學員是否能不經講解就找到當日訓練、看懂內容
- 學員若有不順手回饋給教練，教練再轉述

## 5.5 上線與結案

### 上線狀態

第一版已上線並進入維運。

| 環境 | 狀態                         |
| ---- | ---------------------------- |
| dev  | 持續用於開發 / 整合測試      |
| prod | 已上線，教練與學員實際使用中 |

### 維運模式

| 主題      | 採用做法                                                                       |
| --------- | ------------------------------------------------------------------------------ |
| 監控      | 應用層以結構化 log 寫入；異常透過後台檢視                                      |
| 部署      | 主要分支推送後由 GitHub Actions 觸發 Render Deploy Hook，dev / prod 雙環境分開 |
| 例行清理  | Refresh Token 由背景服務每 24 小時清除過期紀錄                                 |
| VALD 同步 | 由教練手動觸發（非排程），避免無意義的呼叫額度消耗                             |

### 後續迭代輸入

以下項目在第一版 Out of Scope，於 [02-analysis.md § 2.4 範圍界定](02-analysis.md#24-範圍界定-scope) 預留，依教練回饋優先級排入後續。

新需求 / 新變更一律以 5.3 變更管理流程處理。
